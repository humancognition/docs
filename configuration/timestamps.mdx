HCE's Open mode can prefix every encoded identifier with a short timestamp prefix derived from a configurable reference epoch. The prefix serves two purposes at once: it makes the encoded strings lexicographically sortable by creation time, and it makes the rough creation time **audible to a human reader** without any decoding step. This is particularly useful in distributed systems where you need globally unique, pronounceable IDs that also carry ordering and time-bucket signals — without a centralised sequence generator.

## The Audible Timestamp Principle

Conventional time-ordered identifiers like UUIDv7 and ULID embed a timestamp at the bit level. That is fine for machines, but it is invisible to a developer scanning a log line: two adjacent UUIDv7 strings differ in characters that carry no meaning to the eye. HCE inverts this. The timestamp is encoded as two real syllables at the start of the string — for example, `KA-RE` for one time bucket, `KI-RU` for a later one — so a person reading a log can immediately tell that one identifier is older than another without decoding anything.

The mapping is **monotone in time**: the syllable indices increase deterministically with the unit count since the epoch, so two properties hold at once:

- **Auditory ordering.** Reading the prefix aloud, a listener can tell which identifier is newer because the vowel sequence advances predictably.
- **Lexicographic ordering.** A plain string sort produces the same order as a chronological sort, so you can sort encoded identifiers without ever decrypting them.

The body of the identifier is still fully FPE-encrypted, just like in Sealed mode — the prefix is the _only_ thing that reveals creation time. Nothing else about the underlying bytes leaks.

## How timestamp prefixing works

When you call `with_timestamp_config`, you provide two values:

- **`epoch_ms`** — your application's reference epoch in Unix milliseconds. All timestamps are encoded as offsets from this value, which lets you maximise the useful range of the prefix for your deployment era.
- **`granularity`** — how finely to encode time. Coarser granularity means fewer syllables are consumed by the prefix, leaving more room in the output for the encrypted payload; finer granularity provides tighter ordering at the cost of more prefix syllables.

The prefix is always two syllables in the form `K<V>-R<V>` (e.g. `KA-RE`), where each vowel represents a time bucket offset from your epoch at the chosen granularity.

## Granularity options

| Granularity | Typical Use |
| --- | --- |
| `Second` | High-throughput IDs — e.g. event streams, IoT telemetry |
| `Minute` | Most web and API applications |
| `Hour` | Low-frequency IDs — e.g. batch jobs, scheduled tasks |
| `Day` | Date-keyed records — e.g. daily reports, audit logs |
| `Week` | Weekly batch IDs — e.g. subscription billing cycles |
| `Month` | Monthly rollup IDs — e.g. invoices, statement periods |

## Rust example

```rust
use hce_core::{Hce, HceMode, LanguageLevel, TsGranularity};

let key = b"32-byte-secret-key-here-xxxxxx!!";

let hce = Hce::new(Some(key), LanguageLevel::Universal, HceMode::Open)
    .with_timestamp_config(1_700_000_000_000_i64, TsGranularity::Second);

let id = hce.encode(&uuid_bytes);
// The first syllables encode the timestamp prefix, e.g.:
// => "KA-RE-PETREN-NISLORPEN-LAFLER"
//      ^^^^^ timestamp prefix

// Decoding strips the prefix and recovers the original bytes
let bytes = hce.decode(&id).unwrap();
```

The `epoch_ms` value `1_700_000_000_000` corresponds to 2023-11-14 22:13:20 UTC — a common choice for applications launched around that period that want small initial offset values.

<Note>
  Timestamp prefixing only works in `Open` mode. `Sealed` mode does not prepend a timestamp, and configuring `with_timestamp_config` on a `Sealed` or `Plain` instance has no effect on the output.
</Note>

## Sorting by timestamp

Because HCE uses consistent phoneme encoding, two identifiers produced by the same configuration can be sorted lexicographically (case-insensitively) to recover chronological order:

```rust
let mut ids: Vec<String> = events
    .iter()
    .map(|e| hce.encode(&e.id_bytes))
    .collect();

// Lexicographic sort == chronological sort when using Open mode with timestamps
ids.sort();
```

## Python equivalent

```python
import hce

h = hce.Hce(
    key=b"32-byte-secret-key-here-xxxxxx!!",
    level="universal",
    mode="open",
).with_timestamp_config(1_700_000_000_000, "second")

id_str = h.encode(uuid_bytes)
# => "KA-RE-PETREN-NISLORPEN-LAFLER"

original = h.decode(id_str)
```

The `granularity` argument is a lowercase string: `"second"`, `"minute"`, `"hour"`, `"day"`, `"week"`, or `"month"`.

## CLI equivalent

```bash
hce encode \
  --mode open \
  --timestamp-epoch 1700000000000 \
  --timestamp-granularity second \
  --key <hex-key> \
  <hex-input>
```

You can verify sorting behaviour by encoding the same payload at two different times and comparing the outputs:

```bash
# Encode now
hce encode --mode open --timestamp-epoch 1700000000000 \
  --timestamp-granularity minute --key <hex-key> <hex-input>

# Encode one minute later — the leading prefix syllables will differ
hce encode --mode open --timestamp-epoch 1700000000000 \
  --timestamp-granularity minute --key <hex-key> <hex-input>
```

## UUID adapter and UUIDv7

<Tip>
  When working with UUIDv7 identifiers, use the `UuidAdapter` from `hce-adapters` together with Open mode. HCE automatically extracts the 48-bit millisecond timestamp embedded in the UUIDv7 bytes and uses the identifier's own creation time as the prefix source — no separate `with_timestamp_config` call required.
</Tip>

```rust
use hce_adapters::{HceCodec, UuidAdapter};
use hce_core::{Hce, HceMode, LanguageLevel};

let hce = Hce::new(
    Some(b"32-byte-secret-key-here-xxxxxx!!"),
    LanguageLevel::Universal,
    HceMode::Open,
);

// UuidAdapter sets the correct bit width; HCE extracts the UUIDv7 timestamp automatically
let codec = HceCodec::new(hce, UuidAdapter);

let encoded = codec.encode(&uuid_v7_bytes);
// Timestamp prefix is derived from the UUID's embedded timestamp
let decoded = codec.decode(&encoded).unwrap();
```
