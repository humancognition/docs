By default, HCE groups syllables into natural-feeling chunks separated by hyphens — for example, `PETREN-NISLORPEN-LAFLER`. You can override this behaviour with one of three explicit chunking strategies and, independently, swap the separator character for any symbol that does not conflict with the phoneme alphabet. These two knobs let you tune output for human readability, URL safety, database storage, or any other downstream constraint.

## Why HCE Chunks at All

Chunking is not cosmetic in the way phone-number formatting is cosmetic. It is the mechanism HCE uses to fit a 128-bit identifier inside Cowan's [4±1 working-memory limit](https://en.wikipedia.org/wiki/Working_memory#Capacity). Cowan's research on working memory shows that adults reliably hold about four _chunks_ — not characters — in short-term memory at once. A 32-character hex UUID asks a listener to track ~28 unrelated chunks and breaks down over the phone; the same identifier rendered as four or five hyphen-separated syllable groups lands inside that limit, which is why every successful human-friendly identifier scheme — phone numbers (`555-123-4567`), credit cards (`4111 1111 1111 1111`), IBANs, software licence keys — does the same thing.

The default natural chunking is tuned to produce four to five chunks for a 128-bit input. That number is not chosen by trial and error; it is the upper end of what an adult can reliably read back over the phone after a single hearing. Fixed and pattern strategies let you trade off readability against external constraints (URL length budgets, database column widths, branded ID shapes), but for human-channel use cases the natural strategy is almost always what you want.

### Lossless by construction

Two implementation details make chunking safe to use without changing your bytes:

- **Boundaries are derived, not random.** Chunk positions are computed deterministically from the FPE ciphertext bits that have already been counted toward the 128-bit budget. The boundaries read from bits that are "already paid for" — they consume **zero additional entropy**. The same input always produces the same chunk layout, and there is nothing extra to store.
- **Separators are not data.** The decoder strips every separator (hyphens, spaces, line breaks, even mixed or wrong-position separators) before parsing, then recomputes its own internal boundaries. A user can paste the identifier with the original hyphens, with no hyphens, or with hyphens replaced by spaces and the result is identical. This is the same separator-ignoring philosophy used by [Crockford Base32](https://www.crockford.com/base32.html): the canonical form is the syllable sequence, not the rendered string.

### Syllable-aligned boundaries

The natural chunker never splits a syllable. Because a single HCE syllable carries a fixed onset-plus-nucleus pair (with the cluster onset and optional coda treated as one indivisible unit), every chunk boundary falls on a syllable seam. This is what keeps the output pronounceable when read chunk-by-chunk: every chunk is itself a sequence of complete syllables, so a listener can rehearse each chunk through the [phonological loop](https://en.wikipedia.org/wiki/Working_memory) independently before assembling the full identifier.

## Chunking strategies

The four strategies below are mutually exclusive. Set whichever one fits your use case; the last call to `with_chunk_spec` wins.

| Strategy | Method | Description |
| --- | --- | --- |
| Natural (default) | `with_chunk_spec(ChunkSpec::natural_with_separator('-'))` | Groups syllables into word-like chunks using a built-in heuristic, separated by hyphens |
| None | `with_chunk_spec(ChunkSpec::none())` | Produces a single continuous string with no separators |
| Fixed | `with_chunk_spec(ChunkSpec::fixed(n))` | Inserts a separator after approximately every _n_ characters |
| Pattern | `with_chunk_spec(ChunkSpec::pattern(&[3, 4, 4, 3]))` | Applies a repeating cycle of custom syllable counts per chunk |

### Natural chunking

Natural chunking is active by default. You can make it explicit — and change the separator — without any other configuration changes.

```rust
use hce_core::{ChunkSpec, Hce, HceMode, LanguageLevel};

let hce = Hce::new(
    Some(b"32-byte-secret-key-here-xxxxxx!!"),
    LanguageLevel::Universal,
    HceMode::Sealed,
)
.with_chunk_spec(ChunkSpec::natural_with_separator('-'));

let id = hce.encode(&uuid_bytes);
// => "PETREN-NISLORPEN-LAFLER-SRORGULGOLFUN-PREPLEN"
```

### No chunking

Pass `with_chunk_spec(ChunkSpec::none())` to remove all separators and emit a single unbroken string.

```rust
use hce_core::{ChunkSpec, Hce, HceMode, LanguageLevel};

let hce = Hce::new(
    Some(b"32-byte-secret-key-here-xxxxxx!!"),
    LanguageLevel::Universal,
    HceMode::Sealed,
)
.with_chunk_spec(ChunkSpec::none());

let id = hce.encode(&uuid_bytes);
// => "PETRENNISLORPENLAFLERSR0RGULGOLFUNPREPLEN"
```

<Tip>
  Use `ChunkSpec::none()` when persisting HCE strings in a database. The absence of separator characters reduces storage size and eliminates the need for any separator-stripping logic when indexing or querying the column.
</Tip>

### Fixed chunking

`ChunkSpec::fixed(n)` places a separator after approximately every _n_ characters. The engine finds the nearest syllable boundary to keep chunks phonetically clean.

```rust
use hce_core::{ChunkSpec, Hce, HceMode, LanguageLevel};

let hce = Hce::new(
    Some(b"32-byte-secret-key-here-xxxxxx!!"),
    LanguageLevel::Universal,
    HceMode::Sealed,
)
.with_chunk_spec(ChunkSpec::fixed(8));

let id = hce.encode(&uuid_bytes);
// => "PETRNISLO-RPENLAFL-ERSROR-GULGO"  (~8 characters per chunk)
```

### Pattern chunking

`ChunkSpec::pattern` accepts a slice of syllable counts. HCE cycles through the pattern until all syllables are placed. This is useful when you want the output to mimic a specific structure — such as a passphrase word count or a branded identifier format.

```rust
use hce_core::{ChunkSpec, Hce, HceMode, LanguageLevel};

let hce = Hce::new(
    Some(b"32-byte-secret-key-here-xxxxxx!!"),
    LanguageLevel::Universal,
    HceMode::Sealed,
)
.with_chunk_spec(ChunkSpec::pattern(&[3, 4, 4, 3]));

let id = hce.encode(&uuid_bytes);
// => "PET-RNIS-LORP-ENL"  (3 then 4 then 4 then 3 syllables)
```

## Changing the separator character

The separator is set as part of the chunking strategy. For natural chunking, pass your preferred character to `ChunkSpec::natural_with_separator`. It applies to Natural mode; Fixed and Pattern modes default to `'-'` and can be changed by constructing the spec and then setting its `separator` field directly. The separator must not be an alphabetic character or a character used in the HCE phoneme alphabet — hyphens, dots, spaces, and underscores are all safe choices.

<CodeGroup>

```rust Hyphen (default)
let hce = hce.with_chunk_spec(ChunkSpec::natural_with_separator('-'));
// => "PETREN-NISLORPEN-LAFLER"
```

```rust Space
let hce = hce.with_chunk_spec(ChunkSpec::natural_with_separator(' '));
// => "PETREN NISLORPEN LAFLER"
```

```rust Dot
let hce = hce.with_chunk_spec(ChunkSpec::natural_with_separator('.'));
// => "PETREN.NISLORPEN.LAFLER"
```

```rust Underscore
let hce = hce.with_chunk_spec(ChunkSpec::natural_with_separator('_'));
// => "PETREN_NISLORPEN_LAFLER"
```

</CodeGroup>

## Python equivalents

The Python binding exposes chunking and separator control as direct methods on the `Hce` object, matching the ergonomics of the Rust builder pattern.

```python
import hce

h = hce.Hce(
    key=b"32-byte-secret-key-here-xxxxxx!!",
    level="universal",
    mode="sealed",
)

# Fixed chunking — ~8 characters per chunk
fixed = h.with_chunk_fixed(8)
print(fixed.encode(data))

# Pattern chunking — cycle through [3, 4, 4, 3] syllables
patterned = h.with_chunk_pattern([3, 4, 4, 3])
print(patterned.encode(data))

# No separators
compact = h.with_chunk_none()
print(compact.encode(data))

# Change separator to a dot (natural chunking)
dotted = h.with_separator(".")
print(dotted.encode(data))
```

## CLI equivalents

The `hce encode` command exposes all four strategies through dedicated flags.

```bash
# Natural chunking with a custom separator
hce encode --separator '.' --key <hex-key> <hex-input>

# No chunking
hce encode --chunk-none --key <hex-key> <hex-input>

# Fixed chunking (~8 characters per chunk)
hce encode --chunk-fixed 8 --key <hex-key> <hex-input>

# Pattern chunking (comma-separated syllable counts)
hce encode --chunk-pattern 3,4,4,3 --key <hex-key> <hex-input>
```
