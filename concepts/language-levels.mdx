When an HCE string is read aloud, written down, or communicated verbally, the sounds it contains matter. HCE uses different **language levels** to select phoneme sets that feel natural to speakers of different languages. Every level shares the same mathematical structure — 35 onsets, 20 vowel nuclei, radix 700 — so they all produce output of identical length. The only difference is which actual sounds (and their written representations) are used.

## How a Phoneme Set is Chosen

A level is not a hand-curated list. It is the output of two filters applied to a phonological database:

1. **PHOIBLE coverage.** [PHOIBLE](https://phoible.org/) catalogues the phoneme inventories of 3,000\+ documented languages. A phoneme qualifies for a level only if it appears in at least a target percentage of those inventories — 85% for `Universal`, lower thresholds for the more region-specific levels. This is what makes Universal genuinely _universal_: every consonant and vowel in it is a sound that almost every speaker on Earth already produces in their native language.
2. **Sonority Sequencing Principle (SSP).** Consonant clusters are not enumerated by hand. Every candidate cluster is checked against SSP, a cross-linguistic constraint that says sonority must rise toward the vowel peak. The standard sonority scale is:

| Class | Sonority | Examples |
| --- | --- | --- |
| Stop | 1 | `b d g k p t` |
| Fricative | 1.5 | `s f h` |
| Nasal | 2 | `m n` |
| Liquid | 3 | `l r` |
| Glide | 4 | `j w` |
| Vowel | 5 | `a e i o u` |

A cluster is admitted only if the sonority difference between its two consonants meets the level's `MIN_DIST`. `tr` (stop 1 → liquid 3, difference 2) clears easily. `fl` and `fr` (fricative 1.5 → liquid 3, difference 1.5) clear at the relaxed `MIN_DIST = 1.5` used in `Universal`. `pt` (stop → stop, difference 0) fails. `st`, `sp`, and `sk` actually fall in sonority and violate SSP outright — they are dropped from `Universal` and only re-introduced in language-specific levels where speakers tolerate the irregularity.

Tightening either filter — raising the PHOIBLE threshold or relaxing the SSP `MIN_DIST` — produces a different level. That is how `Eu` and `En` are derived: same algorithm, narrower thresholds, different phonemic territory.

## Level Comparison

| Level | Onsets | Vowel \+ Nucleus | Best For |
| --- | --- | --- | --- |
| `Universal` | 35 | 20 | Cross-lingual; phonemes present in ≥85% of PHOIBLE languages |
| `Eu` | 35 | 20 | European language speakers |
| `En` | 35 | 20 | English-only environments |
| `Numeric` | 35 | 20 | Numeric-style text representations |

All four levels use the **same radix** (700 = 35 × 20), so a 128-bit identifier always encodes to the same number of syllables regardless of which level you choose. You can safely switch levels between deployments as long as you re-encode existing identifiers.

---

## Universal

Universal is the recommended default. Its phonemes are drawn from the PHOIBLE inventory at an ≥85% language-coverage threshold — `/m/` (98.8%), `/k/` (94.3%), `/n/` (91.7%), `/s/` (88.2%), `/p/` (85.7%), and so on for the consonants; `/a/`, `/i/`, `/u/`, `/e/`, `/o/` for the vowels. That means a Universal-encoded string is built entirely from sounds that already exist in the listener's native phonology, regardless of whether they are a Mandarin, Hindi, Yoruba, Spanish, or German speaker. Cluster onsets are limited to stop\+liquid and fricative\+liquid pairs that pass SSP, which is why `KRUMA` is pronounceable in nearly every language but `KTUMA` would not appear in HCE output at all.

Single consonant onsets include: `b`, `d`, `g`, `k`, `m`, `n`, `p`, `t`, `s`, `l`, `r`, `f`, `h`, `j`, `w`.

Consonant cluster onsets include: `bl`, `br`, `dr`, `fl`, `fr`, `gl`, `gr`, `kl`, `kr`, `pl`, `pr`, `sl`, `sr`, `tr`, `st`, `sp`, `sk`, `sm`, `sw`, `tw`.

Vowel nuclei are plain vowels (`a`, `e`, `i`, `o`, `u`) and vowel-plus-coda combinations (`an`, `al`, `ar`, `en`, `el`, `er`, `in`, `il`, `ir`, `on`, `ol`, `or`, `un`, `ul`, `ur`).

## Eu

The `Eu` level is tuned for **European language speakers**. It is `Universal` plus the consonant clusters that PHOIBLE shows are widespread in Germanic, Romance, and Slavic inventories but not universal — for example, additional fricative\+stop and liquid\+stop combinations. SSP is still enforced, so clusters that would stall the phonological loop are still excluded. Use `Eu` when you know your audience reads and speaks predominantly European languages and you want to maximise naturalness for that group.

---

## En

The `En` level targets **English speakers** specifically. It is `Eu` plus English-licensed clusters that violate SSP in a strict reading but are well-established in English orthography and speech (`st`, `sp`, `sk`, `sm`, `sw`, `tw`). These clusters are comfortable for English readers but can be mis-syllabified by speakers of languages without `s+stop` onsets, which is why HCE keeps them in a dedicated level rather than promoting them to `Universal`. Choose `En` only when your audience is reliably English-first.

---

## Numeric

The `Numeric` level produces output that **visually resembles numeric codes**. The phoneme representations are chosen to look like digits and short number-like strings rather than natural words. This is useful when you need output that fits visually into a context where users expect something number-like — for example, printed voucher codes or PIN-style identifiers — while still benefiting from HCE's pronounceability and structure.

<Tip>
  Use `LanguageLevel::Numeric` when your product displays identifiers alongside actual numbers and you want a consistent visual style across both. The output still passes through FPE encryption and check syllables — it is just rendered in a numeric-adjacent character set.
</Tip>

## Rust Example

```rust
use hce_core::{Hce, HceMode, LanguageLevel};

let key = Some(b"32-byte-secret-key-here-xxxxxx!!" as &[u8]);
let input: [u8; 16] = [
    0x01, 0x95, 0xe3, 0xa0, 0x7c, 0x2e, 0x7b, 0x41,
    0x8f, 0x3d, 0x9a, 0x6c, 0x1e, 0x0b, 0x4d, 0x27,
];

// Universal — cross-lingual default
let hce_universal = Hce::new(key, LanguageLevel::Universal, HceMode::Sealed);
let id_universal = hce_universal.encode(&input);

// Eu — European language phoneme set
let hce_eu = Hce::new(key, LanguageLevel::Eu, HceMode::Sealed);
let id_eu = hce_eu.encode(&input);

// En — English-only phoneme set
let hce_en = Hce::new(key, LanguageLevel::En, HceMode::Sealed);
let id_en = hce_en.encode(&input);

// Numeric — numeric-style text representation
let hce_numeric = Hce::new(key, LanguageLevel::Numeric, HceMode::Sealed);
let id_numeric = hce_numeric.encode(&input);

// All four strings have the same syllable count — only the
// rendered characters differ. Each codec can only decode its
// own output; levels are not interchangeable.
assert_eq!(
    hce_universal.decode(&id_universal).unwrap(),
    input.to_vec()
);
```

<Note>
  Language levels are encoded in the FPE tweak, so a string encoded at one level cannot be decoded by a codec configured with a different level. If you change the level in production, you must re-encode all existing identifiers.
</Note>
